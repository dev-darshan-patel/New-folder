# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Product

A multi-tenant appointment-booking SaaS (a Calendly-style scheduler) for small businesses. A business owner signs up, sets weekly availability and "event types", and shares a public booking link (`/{businessSlug}/{eventSlug}`) where their customers self-book a time slot. The **core booking MVP**, **Stripe subscription billing**, **self-service reschedule/cancel**, **custom branding + an embeddable widget**, polish features (**.ics calendar invites, invitee timezone selection, owner-side cancel/reschedule**), and **per-event-type intake questions + scheduling limits (min-notice, max bookings/day)** are built. **Email reminders** (24h + 1h before, via a cron endpoint) and a **platform super-admin console** (`/admin`) are also built. Google/Outlook calendar sync and team scheduling are planned and **not built yet** (the user does NOT want SMS).

## Commands

```bash
npm run dev         # Dev server on http://localhost:3000 (Turbopack)
npm run build       # Production build (runs TypeScript checks)
npm run lint        # ESLint (eslint-config-next)
npm run db:migrate  # Apply/create Prisma migrations against the SQLite dev DB
npm run db:seed     # Seed a demo account: demo@demo.com / password123 -> /demo-salon
npm run db:studio   # Prisma Studio GUI for the database
npx tsc --noEmit    # Type-check only
```

After editing `prisma/schema.prisma`, run `npm run db:migrate` to regenerate the client and update the DB.

## Architecture

- **Next.js 16 App Router, all server-first.** Data is read directly in async server components via Prisma; mutations go through **server actions**, not API routes. There is intentionally no REST/tRPC layer. The only client components are the ones that need interactivity (auth forms, `CopyLink`, and the `BookingWidget`).
- **Auth is custom and self-contained — sessions are never delegated to a third party, but Google/Microsoft can authenticate the user.** `src/lib/auth.ts` issues our own signed JWT (`jose`) stored in an httpOnly `session` cookie regardless of how the user authenticated; passwords are bcrypt-hashed (`User.passwordHash` is **nullable** — OAuth-only accounts have none, and `loginAction` rejects password login for them with a redirect-to-the-right-button message). `getCurrentUser()` (request-cached) reads the session. Protected areas guard themselves: `src/app/dashboard/layout.tsx` redirects to `/login` when there is no user. There is no middleware-based auth.
- **"Sign in with Google/Microsoft" is hand-rolled OIDC (Authorization Code flow), not a library, configured at `/admin/settings/auth` (SUPER_ADMIN only) — same DB-backed-credentials pattern as Stripe, not env vars.** `src/lib/oauth.ts` builds the provider authorize URL, exchanges the code for tokens, and **verifies the id_token's signature** via `jose`'s `createRemoteJWKSet` against the provider's real JWKS (not just trusting the userinfo response) — this is the correct, standards-compliant way to do it, not a shortcut. The dynamic routes `src/app/api/auth/[provider]/start` and `.../callback` handle the redirect dance; CSRF is covered by a random `state` embedded in a short-lived httpOnly cookie, compared on callback. `src/lib/oauth-login.ts` `findOrCreateOAuthUser()` resolves the verified profile to a `User`: match by `googleId`/`microsoftId` first, else **link by verified email** to an existing account (so someone who signed up with a password can later also use "Sign in with Google" on the same address), else create a brand-new tenant with the same defaults as email signup (availability + starter event type). An unverified provider email is rejected outright — never auto-linked. Errors and suspended/deleted accounts redirect to `/login?error=...`, mapped to readable text in `src/app/(auth)/oauth-errors.ts`. **Never autonomously flip `PlatformSettings.stripeMode`/equivalent live-vs-test toggles or enter real OAuth credentials — that's the user's call**, same rule as Stripe mode.
- **Multi-tenancy is row-level by `userId`.** Every `EventType`, `Availability`, and `Booking` belongs to a `User` (the tenant). The public site keys off `User.slug`; ownership is always re-checked in dashboard actions (`where: { ..., userId: user.id }`) so one tenant can never touch another's rows.
- **The scheduling engine lives in `src/lib/availability.ts` and is the most delicate code.** Availability is stored timezone-naively as `weekday` + minutes-from-midnight; bookings are stored in **UTC**. `getSlotsForDate` converts a wall-clock time in the business's IANA timezone to a UTC instant using `@date-fns/tz`'s `TZDate`, then filters out past slots and slots overlapping existing bookings. Any change here must preserve the rule: **store UTC, interpret availability in the user's timezone.** Double-booking is prevented both here (slots hide booked times) and again at write time in `createBookingAction` (an overlap re-check before insert).
- **Email degrades gracefully.** `src/lib/email.ts` sends via SMTP only if `SMTP_HOST` is set; otherwise it logs the message to the server console, so the app runs with zero email config. Booking succeeds even if email sending throws.
- **Reminders are pull-based via a cron endpoint, not a background worker.** `/api/cron/reminders` (GET/POST, guarded by `CRON_SECRET`) calls `sendDueReminders()` (`src/lib/reminders.ts`), which emails CONFIRMED bookings entering the 1h and (non-overlapping) 1h–24h windows and stamps `Booking.remind1hSentAt`/`remind24hSentAt` so it's idempotent. A scheduler (Vercel Cron, system cron, etc.) must hit it every few minutes — nothing fires on its own. Email only (no SMS, by product decision).
- **The platform super-admin console lives at `/admin` and is gated by `User.isAdmin`.** `admin/layout.tsx` calls `notFound()` (not redirect) for non-admins so the area's existence isn't revealed. It aggregates across ALL tenants. `isAdmin` is set manually in the DB, never self-serve. This is the only place that intentionally reads across tenant boundaries — keep tenant-scoped queries everywhere else. The `/admin` overview is a full analytics dashboard: **all aggregation lives in `src/lib/admin-metrics.ts` (`getAdminAnalytics(rangeDays)` — one pass over users+bookings)**, rendered with dependency-free SVG charts in `src/components/admin/Charts.tsx` and a 7/30/90-day range via the `?range=` query param. MRR-over-time is approximated from signup dates (no upgrade history is tracked). `/admin/users` is search/filter/sort/paginated entirely via URL query params (parsed once in `src/lib/admin-users-query.ts`, shared with the CSV export route at `/admin/users/export`); `/admin/users/[id]` adds a branding preview, weekly-availability grid, paginated booking history, a Stripe customer link, and internal admin notes (`AdminNote` model, never shown to the tenant) added via `addAdminNoteAction` in `src/app/admin/actions.ts`. **Admin RBAC, audit log, and powerful account actions live under `/admin`, gated by `User.adminRole` (`SUPER_ADMIN`/`SUPPORT`/`READ_ONLY`, nullable = not an admin).** `isAdmin` is a legacy boolean kept only for backward DB compatibility — never read it in new code; `requireAdminRole(min)` in `src/lib/admin-auth.ts` is the authoritative server-side check for every mutating action in `src/app/admin/actions.ts`, and every one of those actions writes a row via `writeAuditLog()` (`src/lib/admin-audit.ts`) to the append-only `AdminAuditLog` table, viewable at `/admin/audit`. **Always enforce role checks server-side in the action, not just by hiding UI** — the UI also hides controls per role for UX, but that's a courtesy layer only. Capabilities: change plan, force a password-reset email (one-time token flow at `/reset-password/[token]`, mirrors the booking `manageToken` pattern), suspend/unsuspend (blocks login in `getCurrentUser()`/`loginAction` and 404s the public booking pages), soft delete (`deletedAt`, hidden from default `/admin/users` list, restorable) and hard delete (SUPER_ADMIN only, requires typing the account's slug to confirm, cascades via existing `onDelete: Cascade`), and **impersonation** (SUPER_ADMIN only — `createSession(userId, { impersonatedBy })` embeds the admin's id in the JWT; `getImpersonator()` reads it back to render the `ImpersonationBanner` in the dashboard layout and power "Stop impersonating"). `/admin/bookings` is a cross-tenant bookings explorer (`src/lib/admin-bookings-query.ts` mirrors the users-list query-param pattern) with search/status/business/date filters, sortable columns, a stats strip (cancellation rate, reschedule rate — tracked via `Booking.rescheduleCount`, kept separate from the ICS `sequence` field since that also bumps on cancel), CSV export, and inline moderation (suspend the booking's business directly from a row, reusing `setSuspendedAction`). A bigger admin build-out is planned in phases — see the approved plan referenced in memory.
- **Event types carry intake questions + scheduling limits, edited at `/dashboard/event-types/[id]`.** `intakeQuestions` is a JSON string (`[{label,required}]`, parsed via `src/lib/intake.ts`); the booking form renders them as `q-{i}` inputs and `createBookingAction` validates required ones and stores answers as JSON on `Booking.answers`. `bufferMinutes` = minimum notice (already used as the slot floor). `maxPerDay` caps bookings per event type per day — enforced in `getSlotsForDate` (a full day returns no slots) **and** re-checked in `createBookingAction`. Always enforce limits/required-answers server-side, not just in the form.
- **Times are stored in UTC and rendered per-viewer.** Slot data carries `startUtc`; the booking page shows a timezone selector (`Intl.supportedValuesOf("timeZone")`, defaulting to the invitee's detected zone) and formats every slot from `startUtc` in the chosen zone. The chosen zone is passed to `createBookingAction` so the invitee's confirmation + the `when` it returns are in their zone, while the owner's email stays in the business zone. The date column is still built in the business timezone (`buildDays`).
- **Booking emails carry an `.ics` invite, and the `Booking.sequence` field exists for this.** `src/lib/ics.ts` builds an RFC-5545 VEVENT (UTC `DTSTART`/`DTEND`); `sendEmail` attaches it. Create sends `METHOD:REQUEST` (sequence 0); reschedule re-sends `REQUEST` and cancel sends `METHOD:CANCEL`, each **incrementing `Booking.sequence`** so calendar clients update/remove the existing event (keyed by UID = the `manageToken`) instead of duplicating it.
- **Invitees self-manage bookings via an unguessable token, not a login.** Each `Booking` gets a random `manageToken` (`crypto.randomUUID()`) embedded in the confirmation email link (`/booking/{token}`). That page and `/booking/{token}/reschedule` let the invitee cancel or move the booking with no account. Cancelling sets `status = CANCELLED`; since slot generation and overlap checks only count `CONFIRMED` bookings, cancelling automatically frees the slot. Reschedule re-checks availability (excluding the booking's own row) before moving it. The `manageToken` is nullable only so pre-existing rows migrated cleanly — always set it on new bookings.
- **Feature entitlements are a code registry + a DB-editable array — `planHasFeature()` is the only read path.** `src/lib/features.ts` defines the closed `FeatureKey` union and `FEATURE_REGISTRY` (label + description per key, rendered as the admin checkbox grid). `Plan.featureKeys: String[]` is the admin-editable source of truth — every plan (FREE/PRO/BUSINESS or any custom tier a SUPER_ADMIN creates at `/admin/settings/plans`) grants an arbitrary subset of the registry, not a fixed per-plan-name bundle. `planHasFeature(planId, key)` in `src/lib/plans.ts` is the single helper every gate in the app must call (never read `featureKeys` or compare `user.plan` against a plan name directly) — call sites span event-type field gating (`dashboard/actions.ts`, intake questions, scheduling limits, video links, redirect/reply-to, approval flow, group/recurring bookings), branding/embed (`src/lib/branding.ts`), team scheduling, CSV export, manual bookings, guest invites, payments eligibility, and calendar busy-sync (enforced at *read time* in `getGoogleBusyWindows()`, not by clearing the tenant's toggle — so a downgraded tenant's stale settings don't keep granting access). **Downgrade rule: existing data/settings from a since-revoked feature keep working; gates only block *creating new* gated things.** The admin plan editor's checkbox grid takes effect immediately on save, with no redeploy. The pricing-page bullet list (`Plan.features: string[]`) is separate, hand-written marketing copy — it is NOT derived from `featureKeys` (some bullets like "Everything in Pro" or "Priority support" aren't real gates), so keep both in sync manually when editing a plan's entitlements.
- **Branding and the embed are plan-gated and resolved server-side.** Brand fields live on `User` (`brandColor`, `brandFont`, `logoUrl`, `welcomeMessage`); `src/lib/branding.ts` `resolveBranding(user, overrides)` returns the effective brand — **but only if `planHasFeature(user.plan, "custom_branding")` is true**, otherwise it falls back to defaults (this is the gate; never apply raw `user.brand*` directly). The public booking page (`/[slug]/[eventSlug]`) reads branding and threads the accent color into `BookingWidget` via inline styles, and loads the chosen Google font via a `<link>`. The **embed** is the same booking page with `?embed=1` (minimal chrome + `EmbedResizer`, which posts document height to the parent). `/public/embed.js` is a dependency-free loader pasted on any site: it injects an inline iframe or a popup, listens for the height message to auto-resize, and reads `data-accent`/`data-font` overrides that become `?accent=&font=` query params (also gated in `resolveBranding`). The embed snippets + live preview are generated in `/dashboard/embed`.
- **Billing is Stripe subscriptions, and `User.plan` is the source of truth.** Plans/limits are defined in `src/lib/plans.ts` (FREE/PRO/BUSINESS by default, plus any admin-created tiers, with `maxEventTypes` gating) — this file is pure product config now, no Stripe credentials in it. Checkout and the billing portal are server actions in `src/app/dashboard/billing/actions.ts`; the only place the DB plan is updated from Stripe is the webhook at `src/app/api/stripe/webhook/route.ts` (verify-signature → sync `plan`, `subscriptionStatus`, `planRenewsAt`). **Never trust the client for plan state — always read `user.plan`.** Feature gates are enforced server-side in actions (e.g. `createEventTypeAction` checks `maxEventTypes`, others check `planHasFeature`), not just in the UI.
- **Stripe credentials are platform-settings, not env vars — configured at `/admin/settings` (SUPER_ADMIN only).** The `PlatformSettings` singleton row (`src/lib/settings.ts` `getPlatformSettings()`) stores **both** TEST and LIVE keys/webhook-secrets/price-IDs at once, plus a `stripeMode` field saying which is active. `src/lib/stripe.ts` `getActiveStripeConfig()` is the only place that resolves "which key to use" — it reads the active mode's fields **with no fallback to the other mode** (switching modes never mixes test and live credentials). `getStripe()`/`isStripeConfigured()`/`getStripePriceForPlan()`/`getPlanForStripePrice()` are now all `async` because of this DB read — every call site (`billing/actions.ts`, `billing/page.tsx`, the webhook route) awaits them. Secret-ish fields (`*SecretKey`, `*WebhookSecret`) are never echoed back into the settings form — the UI shows only `Set — ends in ####`/`Not set` and a separate "Clear" action; `updateStripeSettingsAction` treats an empty secret submission as "leave unchanged" (the input is never pre-filled, so empty really does mean untouched) and never writes key values into the audit log, only which field names changed. `STRIPE_SECRET_KEY`/`STRIPE_PRICE_PRO`/`STRIPE_PRICE_BUSINESS`/`STRIPE_WEBHOOK_SECRET` env vars still work as a zero-config **TEST-mode-only** fallback when the DB field is empty.

## Routing map

- `/` — marketing landing
- `/(auth)/login`, `/(auth)/signup` — route group; the segment is not in the URL
- `/dashboard/*` — protected owner area (overview, event-types, availability, bookings)
- `/[slug]` — a business's public page listing event types
- `/[slug]/[eventSlug]` — the public booking page (renders `BookingWidget`)

Static routes (`/dashboard`, `/login`, `/signup`) take precedence over the dynamic `/[slug]`, so business slugs must not collide with those reserved words.

## Database notes

- Dev uses **SQLite** (`prisma/dev.db`, gitignored). The schema is kept portable; moving to Postgres/MySQL for production means changing the `datasource` provider and re-running migrations. Migrations under `prisma/migrations/` are committed.
- `EventType.slug` is unique **per user** (`@@unique([userId, slug])`), so look-ups use the compound key `userId_slug`.
